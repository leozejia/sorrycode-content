# Agent：从聊天到干活

前面两章讲了 AI 是什么、为什么现在的 AI 不一样了。

但你最想问的问题可能是：

```text
说了半天，Agent 到底跟聊天 AI 有什么区别？
为什么你说它是革命性的？
```

这一章回答这个问题。

<h2 id="scene">一个场景</h2>

你有三份 Excel 报表，分别是 1 月、2 月、3 月的销售数据。你想做一个汇总分析，找出趋势，生成一份 PPT。

**用聊天 AI 的做法：**

你打开第一份 Excel，复制粘贴数据给 AI。AI 分析。你说"继续"，再打开第二份 Excel，复制粘贴。AI 再分析。你重复三次。然后你说"把结果汇总"，AI 给你一段文字。你打开 PowerPoint，照着文字自己做幻灯片。

整个过程里，AI 是顾问，你是操作员。

**用 Agent 的做法：**

你打开工作台，说一句话：

```text
分析这个文件夹里的三份销售报表，找出趋势，生成一份 PPT。
```

Agent 自己打开文件夹，读三份 Excel，分析数据，画图表，生成一份 PowerPoint 文件，保存。

整个过程里，你只做了一件事：下指令。做完后你看了一眼，说"第三页的图表颜色调一下"，Agent 自己改。

```text
用聊天 AI：你做操作，AI 做参谋。
用 Agent：你做指挥，AI 做操作。
```

这就是本质区别。

<h2 id="what-it-is">Agent 是什么</h2>

我们学过，AI 是一个"读过很多书的实习生"。

聊天 AI 是这个实习生被关在一个房间里——你从门缝给他递纸条，他回纸条给你。你们只能文字交流。

Agent 是这个实习生走出了房间。他可以在你公司里走动——看文件、操作电脑、使用工具、执行任务。你跟他的交流从"纸面建议"变成了"实际产出"。

```text
Agent = 模型 + 工具 + 环境 + 循环

模型：大脑，负责理解和决策
工具：手，负责读文件、写文件、执行命令、调用 API
环境：工作空间，通常是你的电脑或项目目录
循环：节奏，指令 → 规划 → 执行 → 反馈 → 调整
```

四个要素里，普通人最需要理解的是"工具"和"循环"。其余两个是技术层的事。

<h2 id="tools">它能做什么</h2>

一个装了 Agent 的电脑，相当于你雇了一个能 24 小时操作电脑的员工。它目前能做的事包括：

```text
读文件：代码、文档、报表、数据库、网页——任何文本文件
写文件：生成文档、代码、配置、报告、幻灯片
执行命令：运行程序、安装软件、启动服务、跑测试
调用工具：发 API 请求、查数据库、操作浏览器
组织信息：整理文件夹、归类文件、汇总多源数据
```

这意味着，你的电脑不再只是一个你手动操作的设备。它变成了一个你下指令的平台。

```text
你 = 总指挥：定方向、给反馈、做判断、签验收
Agent = 执行者：读文件、跑工具、改内容、出结果
```

<h2 id="loop">工作节奏：不是一问一答，是指挥循环</h2>

聊天 AI 的使用方式是：问 → 答 → 问 → 答。每一轮都是独立的。

Agent 的工作方式是：指令 → 规划 → 执行 → 审查 → 迭代。

**第一步：下指令**
你说清楚任务、边界和验收标准。

**第二步：它给计划**
Agent 先不干活。它告诉你它打算怎么做——读哪些文件、用什么工具、产出什么。这是最关键的一步。你在这里拦截错误方向，成本最低。

**第三步：确认后执行**
你点头了，它才开始干活。读文件、跑工具、改内容。

**第四步：你审查结果**
它干完了，告诉你干了什么、改了什么。你检查。不对的地方，你说"这个地方不对，重新做"。

**第五步：迭代**
它根据你的反馈调整，再做一遍。

```text
新手最容易犯的错误：
跳过第二步——直接让 Agent 干活，不看计划。
结果：方向错了，它干了一堆无用功，你还花了钱。
```

这就是为什么前面章节反复强调：你懂的领域你审 AI。你不需要懂它怎么干的，但你得懂它该干什么。

<h2 id="chat-vs-agent">一张表看懂区别</h2>

| | 聊天 AI | Agent |
| --- | --- | --- |
| 怎么用 | 你问，它答 | 你下指令，它干活 |
| 它在哪 | 网页对话框里 | 你的电脑上 |
| 能做什么 | 生成文字 | 读文件、写文件、执行命令、调用工具 |
| 你的角色 | 提问者 | 指挥官 |
| 它有记忆吗 | 每次刷新历史 | 可以记住项目规则（通过配置文件） |
| 适合什么 | 问答、写作、翻译、灵感 | 分析数据、整理文档、改代码、做报表、自动化任务 |
| 不适合什么 | 需要操作文件的任务 | 需要真实人类判断的决策 |

<h2 id="not-magic">它不是魔法</h2>

Agent 很强，但它不是万能的。有三件事你要心里有数：

**1. 它会被困住**

Agent 在遇到意料之外的错误时，可能会反复尝试同一个方法，越试越错。就像新员工卡在一个问题上不知道怎么求助。这时候你需要介入，告诉它换条路。

**2. 它需要好指令**

你对 Agent 说的越模糊，它做的越离谱。"帮我把这个项目做好"——它不知道怎么下手。"分析这三份报表，找出销售额下降的原因，输出一份一页纸的总结"——它能干。

**3. 它不能替代你的判断**

Agent 可以帮你分析数据、生成方案、写报告。但最终决定——这个方案行不行、这份报告能不能发、这个预算该不该批——是你的责任。

```text
Agent 是执行者，不是决策者。
它是你的手和脚，不是你的大脑。
```

<h2 id="why-on-pc">为什么 Agent 要在你的电脑上</h2>

这是这本书的核心论点之一：

```text
你在哪里产出工作，就应该在哪里叠 AI 的 buff。
```

你每天在哪里工作？在你的电脑上。你的文件在电脑上，你的工具在电脑上，你的工作流程在电脑上。

网页 AI 访问不了你的文件，执行不了你的命令，理解不了你的项目结构。它再好，也只能当顾问。

本地 Agent 不一样。它就在你的电脑上，能看到你看到的文件，能用你用的工具。它跟你工作在同一个空间里。

这就是为什么过去一年多，所有主流 AI 厂商都在推本地 Agent——Codex、Claude Code、Cursor，以及国内 DeepSeek、Qwen、豆包、Kimi 各自的 Agent 方案。

```text
不是巧合，是共识。
AI 的价值不在云端，在你的工作环境里。
```

<h2 id="remember">记住一件事</h2>

```text
聊天 AI 给你建议，Agent 替你执行。
你跟聊天 AI 的关系是问答。
你跟 Agent 的关系是上下级——
你指挥，它干活，你审查，它迭代。
```
