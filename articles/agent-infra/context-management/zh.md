# 长期上下文怎么管理

Agent 可以连续工作很久，不代表你应该把所有东西都留在一个会话里。

一开始，长会话很爽。

它记得你刚刚改过什么，知道项目里哪些文件重要，也不用你反复解释同一套背景。很多时候，缓存命中率也会很高，看起来像是“越养越省”。

但这里有个坑：

```text
缓存命中率高，不等于会话状态健康。
```

缓存只说明某些输入被复用了。它不会替你判断这些内容还是不是对的。

如果复用的是项目规则，这是资产。

如果复用的是旧结论、失败尝试、过期路径和临时日志，那就是历史负债。

<h2 id="three-places">先分清三个地方</h2>

长期 agent 工作至少要分三层：

| 放在哪里 | 适合放什么 |
| --- | --- |
| 当前会话 | 正在推进的任务、刚看到的错误、临时判断 |
| 上下文文件 | 长期规则、项目结构、常用命令、稳定偏好 |
| handoff | 释放会话前给下一任 agent 的交接 |

不要把聊天记录当数据库。

聊天记录适合推进，文件适合沉淀，handoff 适合换会话。

<h2 id="context-files">什么该写进上下文文件</h2>

如果一条信息以后还会反复用，就不要每次都重新解释。

写代码，可以放进 [AGENTS.md](/docs/agent-infra/agents-md) 或 [CLAUDE.md](/docs/agent-infra/claude-md)。

做设计，可以放进 [DESIGN.md](/docs/agent-infra/design-md)。

适合长期保存的内容包括：

- 项目是做什么的
- 目录结构
- 常用开发、测试、构建命令
- 代码风格和命名规则
- 重要业务规则
- 设计风格、品牌原则、禁用视觉做法
- 不要改哪些文件
- 完成任务前必须检查什么

这些内容是稳定资产。

<h2 id="do-not-store">什么不要写进长期文件</h2>

不要把这些东西塞进 `AGENTS.md / CLAUDE.md / DESIGN.md`：

- API Key、token、密码
- 一次性任务
- 临时 TODO
- 大段错误日志
- 已经废弃的方案
- 过期路径和过期命令
- agent 自己写错过、后来已经推翻的总结
- 和当前项目无关的长背景

一句话如果不能帮助下一次任务更稳定，就不要写进长期文件。

<h2 id="cache">缓存不是免费的记忆</h2>

Prompt caching 能让重复上下文更便宜、更快，但它不是免费的。

不同模型、不同供应商、不同 runtime 的计费方式会变。新手不用背价格表，只要记住：

```text
输入、输出、缓存输入，可能是三种不同价格。
```

所以不要只看“缓存命中率 90%”。

你还要看：

```text
这些被缓存的内容还在帮我推进任务吗？
```

如果还在推进，就留下。

如果只是把旧噪音反复带进下一轮，就该清掉。

<h2 id="when-release">什么时候该释放会话</h2>

出现这些信号，就不要继续硬养会话：

```text
方向变了
agent 开始引用旧结论
会话里堆了大量日志和失败尝试
你已经说不清它现在到底该听哪条指令
```

这时不要直接关掉。先让 agent 做交接。

<h2 id="handoff">释放前让 agent 写 handoff</h2>

可以复制这段：

```text
我准备释放这个会话了。

请你复核一下当前文档的健康程度，准备一份 handoff 给你的继任者。

这份 handoff 要包括：
- 当前目标
- 已确认的事实
- 已经废弃的方向
- 关键文件和链接
- 下一步只需要看的内容
- 哪些旧信息不要再用

最后给继任者一句话，让他能快速拉齐项目情况。
```

里面最重要的是：

```text
哪些旧信息不要再用
```

很多交接只写“我们做了什么”。真正救命的是“哪些东西已经作废”。

新会话最怕被旧结论污染。你不标出来，下一任 agent 可能会把历史里的每一句话都当成可参考材料。

<h2 id="after-handoff">handoff 之后做什么</h2>

拿到 handoff 之后，按这个顺序处理：

1. 把长期规则写进 `AGENTS.md / CLAUDE.md / DESIGN.md`。
2. 把临时日志、失败尝试和废弃方案留在旧会话里，不要搬过去。
3. 开一个新会话。
4. 让新 agent 先读相关上下文文件和 handoff。

可以这样开始新会话：

```text
请先读取 AGENTS.md / CLAUDE.md 和这份 handoff。

先不要改文件。先告诉我：
- 当前目标是什么
- 哪些结论已经确认
- 哪些旧方向不要再用
- 你下一步准备做什么
```

如果它理解错了，先纠正，再让它继续。

<h2 id="default-rule">默认规则</h2>

长期 agent 工作记住这几句就够：

```text
稳定的，写进文件。
临时的，用完就丢。
复杂的，阶段性交接。
换方向，开新会话。
```

上下文不是越多越好。好的上下文，是能让下一步少走弯路的内容。
