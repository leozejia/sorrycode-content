# 强模型的项目规则怎么写

模型升级后，旧的 `AGENTS.md`、`CLAUDE.md` 和 Skills 不会自动失效。强模型反而可能更认真地执行每一条过时规则，让重复约束、冲突偏好和旧模型补丁一起影响结果。

这不意味着项目规则越少越好。目标是最小正确设计：只保留模型无法从现场可靠推断、又会直接影响正确性的内容。

第一次写规则文件，先看[让 AI 记住项目规则](/docs/agent-memory/remember-rules)。需要了解 Skill 的结构和用途，先看[让 AI 记住工作方法](/docs/agent-memory/remember-skills)。这页只讲模型升级后怎样审计已有规则。

<h2 id="keep">哪些内容应该保留</h2>

| 规则类型 | 为什么保留 |
| --- | --- |
| 项目目标和用户边界 | 模型无法只靠代码判断业务意图 |
| 真源文件和目录职责 | 防止改到生成物、镜像或过期副本 |
| 安装、测试、构建和发布命令 | 让结果可以复现和验证 |
| 数据、权限和外部写入边界 | 限制不可逆或高风险动作 |
| 产品要求和领域规则 | 这些事实不一定存在于公开知识中 |
| 完成标准和证据要求 | 防止没有验证就声明完成 |
| 可靠脚本、模板和工具入口 | 把重复、确定性的执行交给工具 |

这类规则连接的是外部事实和确定性验证。模型能力再强，也不能凭空知道你的生产约束、真源和验收标准。

<h2 id="review">哪些内容应该重新审查</h2>

| 规则类型 | 常见问题 |
| --- | --- |
| 同一偏好的多种写法 | 重复占用上下文，还可能互相冲突 |
| 大段人格和语气设定 | 和当前任务无关，却影响所有输出 |
| 为旧模型补写的逐步操作 | 强模型会机械执行已经不需要的步骤 |
| 无条件自动触发的通用 Skills | 方法论被注入所有任务，压缩现场判断空间 |
| 过多正反例 | 模型开始模仿例子，而不是解决当前问题 |
| 过期路径、命令和工具说明 | 看起来像规则，实际把 Agent 引向旧系统 |
| 要求展示完整思维链 | 可能触发拒绝，也不能替代外部验证 |

不要因为一条规则写得专业就保留它。它需要满足至少一个条件：承载项目事实、约束真实风险，或者修复过可以重复测出的缺口。

<h2 id="minimum-correct-design">最小改动不等于最小正确设计</h2>

“少就是多”是一种设计偏好，不是绝对执行规则。

如果现有方向本身错误，只在原方向上做最小改动，会把错误结构继续保留下来。项目规则可以这样写：

```text
优先选择最小正确设计。

如果现有方向正确，保持改动范围紧凑。
如果现有方向已经错误，先说明证据和目标结构，再完成必要重构。
不要为了减少 diff 继续维护错误抽象，也不要借重构扩大到无关范围。
```

这条规则同时限制两个风险：模型不会为了“最小改动”修补错误方向，也不会把“第一性原理”当成无限重写的授权。

<h2 id="audit">用代表性任务做审计</h2>

不要凭阅读感受一次删掉大半规则。用同一组任务做前后对照。

1. 选出 3 到 5 个真实任务，覆盖日常改动、复杂修复和长程工作。
2. 保存当前基线，记录是否完成、是否漏项、证据是否充分、耗时、Token 和成本。
3. 给规则分组：项目事实、权限、验证、风格偏好、旧模型补丁、工具和 Skills。
4. 每次只删除或合并一组。
5. 用完全相同的任务、输入和完成标准重跑。
6. 结果更好或相同就保留删除；出现稳定退步就恢复最小必要规则。

需要判断的不只是 Token 是否下降，还包括：

```text
交付是否正确
遗漏是否减少
是否引用真实证据
有没有越权动作
有没有无关重构
结果能否通过同一套验证
```

<h2 id="skills">Skills 怎么处理</h2>

Skill 仍然适合承载：

- 特定交付物的稳定流程
- 模型自己拿不到的领域知识
- 需要固定脚本、模板或资产的任务
- 容易出错、必须重复验证的操作

不适合常驻的 Skill：

- 只表达通用工程品味
- 把一种方法当成所有任务的默认答案
- 大量重复 runtime 已经具备的行为
- 没有真实任务验证，却要求所有子 Agent 继承

可选能力默认按需触发。真实缺口反复出现后，再决定是否安装、常驻或扩大适用范围。

<h2 id="template">一份短而完整的项目规则模板</h2>

```md
# Project

这个项目解决什么问题，给谁使用。

## Source of Truth

- 代码真源：
- 内容真源：
- 不要直接修改的生成物：

## Commands

- 安装：
- 测试：
- 构建：

## Authorization

- 可直接执行的本地、可逆动作：
- 需要确认的外部、破坏性或付费动作：

## Design Rule

- 优先选择最小正确设计。
- 现有方向错误时，说明证据后完成必要重构，不保留错误抽象。
- 不扩大到无关范围。

## Verification

- 完成前必须运行：
- 无法验证时必须说明：

## Do Not

- 不提交密钥或私人数据。
- 不修改：
```

模板不是必填表。没有真实内容的段落直接删掉，不要留下空标题。

<h2 id="ask-agent">让 Agent 先做只读审计</h2>

可以复制这段：

```text
请只读审计这个项目中的 AGENTS.md、CLAUDE.md、用户级规则和会自动触发的 Skills。

按下面四类列出每条规则：
- 项目事实
- 权限或风险边界
- 验证要求
- 偏好、重复项或旧模型补丁

每条建议必须说明证据和可能影响。先不要改文件。

然后选出 3 到 5 个代表性任务，设计基线。后续每次只改一组规则，用同一批任务复测。
```

确认审计结果后，再允许 Agent 修改。这样可以避免一次大胆删减把真正有用的方法、脚本和边界一起删除。

<h2 id="model-guides">不同模型还要注意什么</h2>

- Claude Fable 5 强调长程任务、进度证据、自治边界、子 Agent 和记忆，并提醒旧 Skills 可能规定得太细。继续看[Claude Fable 5 怎么用](/docs/runtime/claude-fable-5)。
- GPT-5.6 Sol 强调精简提示词、reasoning effort、工具调用边界和代表性任务评测。继续看[GPT-5.6 Sol 怎么用](/docs/runtime/gpt-5-6-sol)。

<h2 id="default-rule">默认判断</h2>

```text
模型拿不到的事实，写下来。
越权会造成损失的边界，写下来。
结果必须通过的验证，写下来。
模型已经稳定会做的通用判断，不要重复教。
任何删除都用同一组真实任务验证。
```
