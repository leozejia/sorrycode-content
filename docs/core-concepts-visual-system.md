# 核心概念章节视觉系统

更新时间：2026-06-03

这份文档给直接维护 `sorrycode-content` 的运营、内容和生成型 agent 使用。它是内部治理文档，不进入线上 docs。

## 目标

`articles/concepts/` 是 SorryCode 面向新手解释 AI、Agent、Prompt、Context、校准和工具选择的核心教程。配图目标不是制造营销冲击，而是帮助读者建立稳定心智：

- 第一眼知道这是一个成体系的入门课程；
- 每篇封面只承载一个核心隐喻；
- 读者进入文章前先获得方向感，而不是被复杂信息压住；
- 正文解释图再承担具体流程、动作和局部关系。

## 主风格

核心概念章节的封面和正文解释图统一使用：

```text
Open Visual Grammar: whiteboard-operator-sketch
```

使用它的原因：

- 核心概念是给新手建立心智，白板图比杂志封面更像“有人在认真讲清楚”；
- 线团小工适合解释 agent、context、handoff、工具链、token 等操作关系；
- 它能把抽象概念转成一个低科技物理动作，降低理解门槛；
- 白底、手绘、少量中文标注天然适合站内教程，不像营销海报；
- 统一使用这个风格可以形成 SorryCode 自己的教程识别度。

封面和正文图的区别：

- 封面图：更疏、更大留白、一个动作、少量标签，适合在文章卡片和页首快速识别；
- 正文图：可以稍微更具体，承担步骤、对比或局部解释；
- 不要把封面做成完整流程图、架构图或 PPT 页面。

## 封面硬约束

- 文件名：`cover.png`
- 位置：`articles/concepts/<slug>/cover.png`
- 当前生成尺寸：`1536x1024`
- 站内显示安全区：中间 `1536x768`
- 单张图片小于 5 MB
- 不包含 API Key、token、账号、真实订单、内部供应商或私有路由
- 不使用外部图床
- 不依赖图片内长文字表达结论

生成时必须显式传 `--size 1536x1024`。不要用默认方图，不要用后期裁切补救错误比例。重要主体必须完整落在中间 `1536x768` 安全区内。

## 视觉规则

封面应该：

- 纯白或近白背景；
- 一个 GanFan 线团小工；
- 一个低科技物件或路径；
- 一个物理动作讲清一个概念；
- 细黑手绘线条，轻微抖动，保留不完美；
- 少量红、橙、蓝中文手写标注；
- 至少 35% 留白；
- 系列内保持同一主角、线条、色彩角色和信息密度。

封面避免：

- 深蓝商业杂志封面；
- 巨型标题压满画面；
- 随机英文杂志元素、条码、期刊编号；
- 复杂 UI 面板、数据看板或代码截图；
- 正式架构图或流程图；
- 过密的信息图；
- 一张图里塞进多个概念；
- 生成模型不稳定的长中文文字；
- 可爱贴纸化、吉祥物化或表情夸张的小工。

## 文本处理

默认不让图片生成器绘制文章标题。标题由站内页面负责。

封面允许出现 2 到 6 个短中文标签，但必须满足：

- 不超过 1 到 3 个词；
- 不是文章主标题；
- 即使标签生成失败，画面仍能成立；
- 提交前必须逐字目检。

## 首批样张

先做两张样张，不直接替换全组：

### `what-is-ai`

文章核心：

```text
AI 不是神秘魔法，也不是数据库。它像一个读过很多书但没有真实记忆的新员工。
```

推荐隐喻：

```text
线团小工站在一本巨大打开的书旁，把书页里冒出的“模式碎片”装进一个小工具箱。旁边有一张空工牌，提醒它不是神，不是数据库，而是需要被分配任务的新员工。
```

第一眼：

```text
AI 读过很多，但要干活仍然需要人给它任务和边界。
```

### `what-is-context`

文章核心：

```text
上下文是 AI 回答前能看到的所有东西。它像一张工作台，必须区分墙上规则、桌上当前任务和垃圾桶里的旧信息。
```

推荐隐喻：

```text
线团小工整理一张白板工作台：墙上贴“规则”，桌上放“当前任务”，旧纸被推入“垃圾桶”，只有有用资料被放进“上下文盒子”。
```

第一眼：

```text
不是越多越好，而是要把 AI 面前的工作台整理干净。
```

## 评审标准

每张样张至少检查：

- 是否一眼能看出主隐喻；
- 是否像同一组教程封面；
- 是否白底、手绘、线团小工、低科技动作成立；
- 是否没有假字、乱码、条码、期刊编号或随机 logo；
- 是否原生生成 `1536x1024`，且中间 `1536x768` 安全区保留主体；
- 是否适合长期放在站内核心教程里。

## 生成与存储流程

1. 先在媒体临时区生成候选，不进入公开文章目录。
2. 确认方向后，把最终图复制到对应文章目录的 `cover.png`。
3. 不把请求日志、响应日志、失败候选图或生产 prompt 放进 `articles/`。
4. 如果需要保留生成方法，写成内部文档或 OVG pattern，不写成公开文章内容。
5. 提交前运行 JSON 和图片大小检查。
